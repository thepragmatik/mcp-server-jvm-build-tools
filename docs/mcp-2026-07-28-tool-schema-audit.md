# MCP 2026-07-28 RC — Tool JSON Schema 2020-12 Audit (SEP-2106)

This document records the audit performed for issue **#86** and the enforcement
shipped alongside it. It is grounded in inspection of the actually-resolved
dependency artifacts and the real schemas generated at runtime — not assumptions.

## 1. What the RC requires

From the official changelog
(https://modelcontextprotocol.io/specification/draft/changelog, "Minor changes"):

> "Loosen `inputSchema` and `outputSchema` to allow any JSON Schema 2020-12
> keywords, and `structuredContent` to allow any JSON value. Add `$ref`
> resolution requirements and composition-keyword resource bounds (SEP-2106)."

From the Tools page
(https://modelcontextprotocol.io/specification/draft/server/tools):

- `inputSchema` — JSON Schema defining expected parameters. **Defaults to 2020-12
  if no `$schema` field is present** and MUST be a valid JSON Schema object (not
  `null`).
- Implementations must **follow the `$ref` resolution requirements**, must **not
  auto-dereference external `$ref` URIs**, and should **bound schema depth and
  validation time** (composition-keyword resource bounds).

## 2. Where this server's schemas come from

Every tool is declared with `@Tool` / `@ToolParam` and registered via
`MethodToolCallbackProvider` in
[`BuildToolsApplication`](../src/main/java/com/pragmatik/buildtools/BuildToolsApplication.java).
Each tool's `inputSchema` is therefore **auto-generated by Spring AI
`2.0.0-RC2`** — the application contains no hand-written tool schemas.

## 3. Recorded finding: the emitted dialect is already JSON Schema 2020-12

Inspection of the resolved `spring-ai-model:2.0.0-RC2` artifact
(`org.springframework.ai.util.json.schema.JsonSchemaGenerator`) shows that
`generateForMethodInput(...)`:

- generates with the victools schema generator configured for
  `com.github.victools.jsonschema.generator.SchemaVersion.DRAFT_2020_12`;
- explicitly stamps the document root with
  `"$schema": "https://json-schema.org/draft/2020-12/schema"`;
- hoists any generated `$defs` to the document root
  (`JsonSchemaUtils.hoistDefsToRoot`);
- sets `additionalProperties: false` on generated object schemas.

A representative schema captured at runtime from a registered tool:

```json
{
  "$schema" : "https://json-schema.org/draft/2020-12/schema",
  "type" : "object",
  "properties" : {
    "projectDir" : { "type" : "string", "description" : "Path to the project directory" },
    "command"    : { "type" : "string", "description" : "Build command to execute" }
  },
  "required" : [ "projectDir", "command" ],
  "additionalProperties" : false
}
```

**Conclusion:** the dialect emitted by Spring AI RC2 is already JSON Schema
2020-12. There is **no draft-07 divergence and no migration path is required.**

## 4. `$ref` resolution and resource bounds

- The 28 currently registered tools take flat, string-typed parameters. **None of
  them emit `$ref`, `$defs`, or composition keywords (`oneOf`/`anyOf`/`allOf`)**,
  so there is no external-`$ref` dereferencing in the current surface.
- To keep this true as tools evolve, the enforcement test treats only **local
  fragment references** (`#...`) as permitted and fails on any external `$ref`
  URI (`http(s):`, `file:`, `urn:`, relative document references).
- Schema nesting depth and `$defs` count are bounded, and meta-schema validation
  runs under an explicit time budget, satisfying the SEP-2106
  composition-keyword resource-bound requirement.

## 5. Enforcement

[`ToolJsonSchemaComplianceTest`](../src/test/java/com/pragmatik/buildtools/ToolJsonSchemaComplianceTest.java)
boots the application context, enumerates every registered tool, and for each
generated `inputSchema` asserts that it:

1. is a non-null JSON **object** with a root `type: "object"`;
2. declares the 2020-12 dialect when a `$schema` keyword is present (and the
   server always stamps it);
3. validates against the official **JSON Schema 2020-12 meta-schema** (bundled
   classpath resource, no network dereference) within a bounded time budget;
4. compiles cleanly under the 2020-12 dialect, proving every local `$ref`
   resolves;
5. uses **only local** `$ref` fragments — never an external URI;
6. respects the schema depth and `$defs` count bounds.

The meta-schema, registry, and JSON Schema 2020-12 dialect come from
`com.networknt:json-schema-validator`, already a compile-scope dependency on the
MCP stack (via `mcp-json-jackson3`).

## 6. Backward compatibility

This change is **test- and documentation-only**. It adds no runtime code, changes
no tool definitions, and alters no wire behaviour, so it is fully backward
compatible with existing MCP clients. It is a guardrail that fails the build if a
future change (or a Spring AI upgrade) emits a non-conforming schema.

## 7. Out of scope (optional follow-on)

No tool currently declares an `outputSchema` / `structuredContent` contract
(structured results are returned as JSON text via helpers). Adopting
`outputSchema` for high-value structured tools is a separate, optional follow-on
and is intentionally not part of this change.
